Who uses it

User stories for humans and agents.

The overlay is most valuable when somebody needs to answer “what is this page structurally?” fast, without opening DevTools or hand-writing selectors.

For humans.

Designers, devs, QA, content editors.
  • ·As a QA lead, I want to audit a production page I do not own, so I can confirm landmark, heading, and focus structure in under a minute.
  • ·As a content editor, I want missing-alt images called out instantly, so I can fix them before the page ships.
  • ·As a designer, I want to see the real heading hierarchy on the canvas, so I can catch structural drift without reading DOM trees.
  • ·As a frontend engineer, I want repeating cards grouped visually, so the broken sibling in a grid is obvious on first glance.
  • ·As an accessibility reviewer, I want tab order numbered in-place, so I can see keyboard flow without tabbing through the whole page.
  • ·As a release owner, I want the overlay to tear down cleanly, so I can inspect live production pages without leaving residue behind.

For agents.

Coding agents, QA bots, scrapers, test generators.
  • ·As a browser-driving agent, I want a single injected script that exposes page structure visually, so I can reason from annotated screenshots instead of raw DOM soup.
  • ·As a Playwright worker, I want deterministic layer toggles like toggle('heading'), so I can capture one clean screenshot per concern.
  • ·As a scraper agent, I want repeating-component clusters surfaced without selectors, so I can lock onto the right list or card grid automatically.
  • ·As a CI regression agent, I want focus order and heading jumps made visible, so broken keyboard flow shows up as a screenshot diff.
  • ·As an orchestration agent, I want to inject the overlay into a sandboxed preview and control it over postMessage, so parent and child frames stay decoupled.
  • ·As a production-safe automation harness, I want overlay chrome isolated from the host DOM, so the annotations never get mistaken for app state.

How it works

Inject, detect, paint, toggle.

The script is intentionally small. It does four things, and each maps directly to a deployment surface a human or agent can understand.

01 · Inject

Load one file.

Use a bookmarklet, a plain <script> tag, or page.addScriptTag(). Installation creates a single fixed host node at the document root.

02 · Detect

Scan visible structure.

The overlay walks visible landmarks, headings, focusables, missing-alt images, repeating siblings, nesting depth, and block-level layout boxes.

03 · Paint

Render in a shadow root.

Outlines, badges, toolbar, and help chrome live inside an open shadow root, so page CSS cannot leak in and teardown stays clean.

04 · Repaint on change

Stay in sync.

scroll, resize, and MutationObserver events funnel into one RAF-throttled render loop, so SPA updates and DOM churn stay aligned.

Agent interface

One global, six controls.

After injection, the overlay exposes window.__a11yOverlayInstalled. Five controls live on that object; the sixth is a parent-frame postMessage bridge for iframe demos and orchestrators.

.toggle(key)Toggle any layer by name: 'landmark', 'heading', 'interact', 'alt', 'repeat', 'focus', 'depth', 'grid'.
.stateLive state object. Read which layers are active. Inspect after toggling.
.toggleHelp()Show/hide the help panel. Useful for clean screenshots.
.render()Force a repaint. Handy after page.evaluate-driven layout changes.
.teardown()Remove every overlay node, detach observers, restore the page to byte-exact original state.
postMessageFrom a parent frame: iframe.contentWindow.postMessage({__a11yov:true, key:'L'}) — drives an overlay inside a sandboxed iframe.
playwright · typescript · full flow 
// 1. load the page
await page.goto('https://example.com');

// 2. inject the overlay
await page.addScriptTag({ path: 'a11y-overlay.js' });

// 3. drive it — one screenshot per layer
for (const layer of ['landmark', 'heading', 'alt', 'focus']) {
  await page.evaluate((k) => {
    const api = window.__a11yOverlayInstalled;
    // turn everything off, then one on
    Object.keys(api.state).forEach(n => api.state[n] = false);
    api.toggle(k);
  }, layer);
  await page.screenshot({ path: `audit-${layer}.png` });
}

// 4. clean up — host DOM is byte-exact
await page.evaluate(() => window.__a11yOverlayInstalled.teardown());

Under the hood

Invisible to the page. Visible to you.

The overlay is engineered to be safe on production: nothing you see on top of your page can ever be confused with your page.

Shadow-DOM chrome.

All UI — toolbar, help panel, outlines — lives inside an open shadow root attached to a single fixed-position host. Your stylesheets can't leak in, and your selectors can't accidentally match overlay nodes.

Byte-exact teardown.

Press X (or call teardown()) and every trace is gone. No class residue, no inline styles, no leftover listeners. document.body.innerHTML matches what it was before install.

RAF-throttled rendering.

Scroll, resize, and DOM mutations all funnel into a single requestAnimationFrame tick. Painting 500 outlines on a long page stays at 60fps on a mid-range laptop.

MutationObserver-aware.

SPA route changes, React re-renders, HTMX swaps — the overlay follows. Overlay-internal mutations are filtered out, so there's no render loop with your own observers.

Cross-frame remote control.

Inject into an iframe, drive it from the parent via postMessage. Useful for design-review apps, component galleries, and agent orchestrators that surface a preview UI.

Keyboard-first, mouse-optional.

Every layer has a single-letter toggle. The floating toolbar mirrors the keys for mouse users. Keyboard listeners never hijack typing inside inputs or contentEditable regions.

Questions

Edge cases, honestly.

Is this a full WCAG scanner?

No — and saying so is a feature. a11y-overlay visualises structure: landmarks, heading order, interactive targets, alt presence, tab order. It does not check contrast ratios, ARIA attribute validity, or focus-visible state. Pair it with axe-core or Lighthouse for exhaustive compliance; reach for a11y-overlay when you need to see the page's skeleton in under a second.

Where can I use it today?

Four places. On pages you own, drop in <script src="/a11y-overlay.js">. On friendly third-party pages, use the bookmarklet. On CSP-heavy sites, use the browser extension. In automation and agents, inject it with Playwright or Puppeteer. What you cannot do with a bookmarklet is bypass a strict Content-Security-Policy; sites like GitHub, Gmail, Stripe, and many admin panels will block script injection by design.

Can an agent extract structured data, not just screenshots?

Today: yes, via window.__a11yOverlayInstalled.state and by querying the shadow DOM directly. A first-class getReport() method — returning { altMissing, headings, focusOrder, repeats } as JSON — is on the short-term roadmap. File an issue if you want to shape the schema.

Does it work on cross-origin iframes?

No, and there's no way around this — the browser's same-origin policy prevents any script injection or DOM access across origins. Inject the overlay inside the iframe's own page (server side, a script tag, or a browser extension), then drive it from the parent via postMessage.

Will it slow down a large page?

Rendering is RAF-throttled and proportional to visible elements, not total DOM size. A 10,000-node page with a 200-node viewport paints in a single frame. The MutationObserver filters overlay-internal changes to avoid feedback loops.

Is it safe to ship in production?

The script is safe — zero host DOM mutation, byte-exact teardown. That said, we recommend gating it behind a cookie, query param, or role check. You probably don't want every visitor seeing outlines.

Why single-letter keys? Won't they conflict?

Keys fire only when no input/textarea/contentEditable is focused, and only on bare presses (no Cmd/Ctrl/Alt). If your page already binds L, disable the overlay's listener by setting window.__a11yOverlayInstalled = null and calling .render() via your own UI.
