Buyer

Start with the team already paying for flaky, under-explained runs.

The primary buyer is not every accessibility team and not every frontend group. The clean wedge is automation or platform teams running Playwright and browser agents who need execution context plus post-run diagnostics.

Primary buyer

Automation and platform teams

They already own browser execution, already feel the cost of vague failures, and already have budget for tools that reduce triage time without making tests less deterministic.

  • Job: keep browser runs reliable when the product surface is dynamic, repeated, and stateful.
  • Pain: the DOM is too noisy, screenshots are too opaque, and failures arrive with weak evidence.
  • Win: one runtime layer for planning, waiting, asserting, and handing off proof.
  • Accessibility teams are a secondary expansion. Missing alt text, heading order, target sizing, and evidence bundles fit naturally, but they are not the clearest first buyer story.
  • Frontend engineers benefit after the wedge lands. Runtime structure and bundles make refactor verification better, but that is adjacent to the execution problem.
  • Regression triage is where the value compounds. Once runs keep their own context and proof, red builds get shorter to explain and cheaper to reopen.
Difference

Raw DOM and screenshots both leave browser automation half blind.

The product is strongest when framed as the layer between those two extremes: enough structure to guide action, enough evidence to explain what happened after.

Raw DOM only

Detailed but noisy

The page exposes everything, but the agent still has to guess which parts matter now.

  • Weak ranking of likely targets in repeated layouts
  • Fragile against style and class churn
  • Little help deciding when the page is truly ready
Screenshots only

Readable but opaque

The run looks plausible to a human, but structure and queryable state disappear.

  • No direct semantic query surface
  • Weak for deterministic assertions
  • Hard to distinguish cosmetic change from structural drift
Overlay runtime

Readable and queryable

Structure becomes shared language for planning, waiting, debugging, and handoff.

  • Action planning grounded in semantics
  • Ready state exposed before and after interaction
  • Artifacts that explain failures instead of only recording them
Run loop

The loop is simple: inspect, wait, act, explain.

The product gets real when it fits inside an existing Playwright workflow. No one needs another dashboard if it does not improve the run itself.

  1. 01

    Open a real target or a deterministic fixture

    Use the same runtime for development fixtures, CI smoke flows, and brittle production surfaces.

  2. 02

    Connect the semantic layer

    Expose landmarks, headings, repeated groups, interactive targets, and diagnostics before the next action is chosen.

  3. 03

    Wait on rendered and idle state

    Replace selector sleeps and screenshot polling with a state model that actually describes the page.

  4. 04

    Keep the evidence bundle

    When a run fails, the page already produced the diagnostics, visuals, and summaries needed for review.

Contract

A small runtime contract is the right product boundary.

The visual overlay helps people. The durable asset is a compact execution surface that agents and test code can trust without reading the whole DOM tree by hand.

The contract should stay tight: state, findings, ready signals, export paths, and teardown. That is enough to guide action and defend failure review.

  • State: read active lenses, mode, and session context.
  • Diagnostics: counts, findings, summaries, and actionable rollups that can travel into reports.
  • Ready signal: wait for rendered or idle state instead of sleeping and hoping the page settled down.
  • Evidence export: keep one artifact bundle with report data and viewport proof.

Proof

The product should make review shorter, not just more visual.

The value compounds when the same runtime supports planning before the click and explanation after the run. That is where the evidence bundle and accessibility overlays meet.

What it is

An execution layer for browser automation with semantic inspection, stable waiting, and exportable evidence.

  • Runtime context for Playwright and agents
  • Built-in accessibility diagnostics
  • Artifacts ready for handoff and review

What it is not

Not a generic AI shell, not a screenshot annotator alone, and not a replacement for deterministic tests.

  • Does not replace product-specific assertions
  • Does not turn flaky flows into magic
  • Does not need a heavy control plane to justify itself
